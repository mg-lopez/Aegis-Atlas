Aegis Atlas - Hackathon Development Plan

Objective
Deliver a convincing hackathon demo quickly: a map-based web dashboard that shows location-specific hazard intelligence generated from multiple signals, without over-optimizing for production concerns.

Phase 1 - Demo Stability (Fastest ROI)
1. Make the pipeline deterministic and test-green across local/runtime environments.
2. Normalize filesystem paths (replace hardcoded container-only paths with project-relative/configurable paths).
3. Fix interpreter portability in tests and demo commands.
4. Ensure one-command demo run is reliable.

Phase 2 - MVP Product Surface (Showcase UI)
1. Build a simple web dashboard with a world map as the primary interface.
2. Add location input (manual coordinates + browser geolocation fallback).
3. Display threat level, confidence, and concise recommended actions for the selected location.
4. Show source summary and last-updated timestamp.

Phase 3 - Multi-Source Agent Logic (Demo Credibility)
1. Keep Sentinel-2 analysis as one signal.
2. Integrate at least two hazard feed adapters (for example GDACS + a seismic/weather/government alert feed).
3. Implement weighted signal fusion to output threat_level, confidence, and rationale.
4. Produce explainability payloads that the UI can render ("why this alert").

Phase 4 - Hackathon Polish
1. Add scripted demo scenarios with known coordinates (low/medium/high).
2. Auto-generate debug artifacts and snapshots for presentation backup.
3. Create a concise 3-minute demo runbook and fallback flow.

Phase 5 - Stretch Goals (If Time Remains)
1. Add basic notification delivery (email/webhook).
2. Support multi-location/family watchlists.
3. Add simple history persistence for recent alerts.

Execution Approach
1. Complete phases sequentially.
2. Validate each phase with runnable commands/tests before moving on.
3. Prioritize demo reliability, explainability, and visual clarity over production hardening.
